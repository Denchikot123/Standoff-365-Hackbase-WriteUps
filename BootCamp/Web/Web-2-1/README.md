# [Web 2-1] LFI на узле www.edu.stf

## Описание

Реализуйте LFI на узле www.edu.stf (10.124.1.235).

Флаг лежит в файле /etc/pt.flag.

## Расследование цели

По условию, мы знаем, что в системе уже должна быть уязвимость **LFI - уязвимость, позволяющая читать/запускать локальные файлы сервера через эндпоинт.**

Откроем вэб ресурс в браузере и перехватим запросы в **Burp Suite.**

<img width="503" height="99" alt="изображение" src="https://github.com/user-attachments/assets/88f8f6ba-6dd3-4dcf-b94d-b46164ad95f4" />

Нам пришло 2 запроса:
* Первый запроc: Сама страница на сайте
* Второй запрос: Судя по эндпоинту (/api/read.php?file=) можно предположить что идет чтение каких то файлов, и так как изначальный путь указан **явно** /opt/cpu.txt, то это файлы из сервера.

## Реализация LFI

Попробуем отправить запрос в Repeater и поменять в запросе путь к файлу и прочесть базовый системный файл **/etc/passwd**.

<img width="412" height="64" alt="изображение" src="https://github.com/user-attachments/assets/95db13f7-31e0-4d21-ab10-44de36bbd008" />

После рефакторинга запроса отправим и посмотрим на респонс

Сервер дал нам прочесть системный файл, что дало нам прямое понимание наличия LFI в этом эндпоинте.

По условию задачи нам необходимо прочесть файл **/etc/pt.flag**

<img width="390" height="177" alt="изображение" src="https://github.com/user-attachments/assets/afe6da69-1dc5-4199-a15e-45382c80c3ff" />

Мы получили флаг путем эксплуатации уязвимости **LFI**

> **Устранение уязвимости:**  
Для защиты от LFI необходимо отказаться от прямой передачи путей к файлам через пользовательские параметры. Рекомендуется использовать белый список разрешенных файлов или использовать жестко заданные идентификаторы (ID), которые внутри кода сопоставляются с конкретными файлами.
